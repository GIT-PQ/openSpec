# PRD：医疗器械专利分类系统功能增强

## 1. 背景与目标

### 1.1 问题

论文评审第10条意见指出：

> 第五章，需求系统功能完整性不足，仅实现单条专利文本分类，未支持批量上传、批量分类、结果导出等实际业务高频，工程性较弱。未表述系统的非功能需求。整体流程缺少"系统测试与验证"，开发流程不完整。

### 1.2 现状

| 功能 | 前端 | Java后端 | Python服务 |
|------|------|----------|-----------|
| 单条分类 | ✅ | ✅ | ✅ |
| 分类历史持久化 | ❌ | ❌ | — |
| 历史记录展示与筛选 | ❌ | ❌ | — |
| 结果导出 | ❌ | ❌ | — |
| 批量导入分类 | ❌ | ❌ | ❌ |

### 1.3 目标

补齐系统功能短板，使系统具备实际业务可用性，同时为论文第五章提供完整的工程化论述素材。

---

## 2. 改进流程与依赖关系

```
步骤1: 持久化(存)  →  步骤2: 展示+筛选(看)  →  步骤3: 条件导出(取)  →  步骤4: 批量导入(进)
        │                    │                        │                       │
        ▼                    ▼                        ▼                       ▼
   分类结果自动存库      历史页面+筛选条件        筛选后一键导出Excel      上传文件+批量推理
                                                                        ↓
                                                                 数据回流入"看"
```

**核心原则**：步骤1完成后，现有单条分类功能自动产生历史数据，步骤2上线即有数据可看，无需等待批量导入。

## 3. 步骤1：分类结果持久化

### 3.1 需求描述

每次执行专利分类后，将分类结果自动存入数据库，包括输入文本、预测类别、类别索引、置信度、全量概率分布等信息。存库为附加能力，存库失败不影响用户获取分类结果。

### 3.2 数据库设计

新增 `classification_record` 表：

| 字段名 | 类型 | 长度 | 约束 | 说明 |
|--------|------|------|------|------|
| id | bigint | 20 | PK, AUTO_INCREMENT | 记录唯一标识 |
| user_id | int | 11 | NOT NULL, INDEX | 操作用户ID，关联user表 |
| summary | text | — | NOT NULL | 输入的专利摘要文本 |
| pred_label | varchar | 100 | NOT NULL | 预测的一级类别名称 |
| pred_index | int | 11 | NOT NULL | 预测类别索引(0-21)，对应LABELS数组下标 |
| pred_probability | double | — | NOT NULL | 预测类别的置信度(0~1) |
| top_categories | json | — | NULL | 全部22个类别概率分布，JSON格式 |
| source | varchar | 20 | NOT NULL, DEFAULT 'single' | 来源：single(单条输入)/batch(批量导入) |
| batch_id | varchar | 50 | NULL, INDEX | 批次ID，批量导入时关联同一批次 |
| create_time | datetime | — | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 分类时间 |

`top_categories` JSON 格式示例（存全部22个类别，按概率降序排列）：

```json
[
  {"name": "中医器械", "probability": 0.8532, "index": 0},
  {"name": "有源手术器械", "probability": 0.0401, "index": 14},
  {"name": "医用成像器械", "probability": 0.0298, "index": 3},
  ...
]
```

### 3.3 user_id 获取方式

前端请求拦截器（`api/index.js`）已在所有请求头中自动附加 `Authorization: Bearer token_{userId}_{timestamp}`，后端 `UserController.getUserInfo()` 已有从 token 解析 userId 的逻辑。步骤一在 `PatentController.classify()` 中复用相同的 token 解析逻辑即可获取 user_id，**前端零改动**。

### 3.4 存库策略

分类是核心功能，持久化是附加能力。存库失败时仅记录日志，不影响用户获取分类结果：

```
Python /predict 成功
       │
       ▼
PatentServiceImpl 收到结果
       │
       ├─ 提取 pred_label, pred_index, pred_probability, categories
       ├─ 构造 ClassificationRecord 对象
       ├─ try { recordService.save(record) }
       │   catch { log.error("分类记录存库失败", e) }  ← 静默失败
       │
       └─ 返回分类结果给前端（不受存库影响）
```

### 3.5 改动范围

| 层 | 改动点 | 说明 |
|----|--------|------|
| 数据库 | 建表SQL | 新增 `classification_record` 表 |
| Java后端 | 新增 `ClassificationRecord` 实体 | 含全部字段（含 pred_index，不含 ip_address） |
| Java后端 | 新增 `ClassificationRecordMapper` | MyBatis接口+XML映射 |
| Java后端 | 新增 `ClassificationRecordService` | 存库逻辑（insert方法） |
| Java后端 | 修改 `PatentController.classify()` | 从 Authorization 头解析 userId，传入 service |
| Java后端 | 修改 `PatentServiceImpl.classify()` | 分类成功后尝试存库，失败仅记日志 |
| Python服务 | 无改动 | — |
| 前端 | 无改动 | 请求拦截器已自动带 token |

### 3.6 验收标准

- 单条分类成功后，`classification_record` 表中自动新增一条记录
- 记录内容包含完整的输入摘要、预测结果、类别索引、置信度和全部22个类别的概率分布
- user_id 从请求头 token 中正确解析
- 分类失败（Python服务返回非200）时不产生记录
- 存库失败时用户仍能正常获取分类结果，后端日志记录错误信息

## 5. 步骤3：条件导出

### 5.1 需求描述

在历史记录页面，支持将筛选后的记录导出为Excel文件，包含摘要文本、预测类别、置信度、分类时间等字段。

### 5.2 交互设计

```
┌─────────────────────────────────────────────────────────┐
│  筛选区域                                                │
│  [预测类别▼] [时间范围] [来源▼] [摘要关键字___]            │
│  [查询] [重置]                                             │
│                                           [📥 导出Excel] │
└─────────────────────────────────────────────────────────┘
```

点击"导出Excel"按钮：以当前筛选条件查询全量匹配数据，生成Excel文件并触发浏览器下载。

### 5.3 API 设计

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/record/export` | 按筛选条件导出Excel |

**请求参数**：与 `/api/record/list` 相同的筛选参数。

**响应**：`Content-Type: application/octet-stream`，返回 `.xlsx` 文件流。

### 5.4 导出Excel字段

| 列名 | 对应字段 | 说明 |
|------|----------|------|
| 序号 | — | 自增序号 |
| 专利摘要 | summary | 完整文本 |
| 预测类别 | pred_label | 一级类别 |
| 置信度 | pred_probability | 百分比格式，如85.23% |
| 来源 | source | 单条输入/批量导入 |
| 批次ID | batch_id | 批量导入时显示 |
| 分类时间 | create_time | yyyy-MM-dd HH:mm:ss |

### 5.5 改动范围

| 层 | 改动点 | 说明 |
|----|--------|------|
| Java后端 | pom.xml 新增 EasyExcel 依赖 | 阿里巴巴EasyExcel，轻量高效 |
| Java后端 | `RecordController` 新增 `export` 方法 | 查询数据+生成Excel+流式响应 |
| Java后端 | `RecordService` 新增 `queryAll` 方法 | 全量条件查询（隐式过滤userId） |
| 前端 | `ClassificationHistory.vue` 新增导出按钮 | 调用导出接口，触发文件下载 |

### 5.6 细化决策

| 决策点 | 结论 | 说明 |
|--------|------|------|
| Excel库选型 | EasyExcel | 阿里巴巴出品，兼容性好，API简洁 |
| 导出上限 | 200条 | 超出时前端提示用户缩小筛选范围 |
| batch_id空值处理 | 占位符 "-" | 单条记录无批次ID时显示 "-" |
| 前端下载实现 | responseType:'blob' + createObjectURL | axios 设置 responseType，创建临时 URL 触发下载 |
| Content-Type | 标准 MIME type + Content-Disposition:attachment | 使用 `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`，设置 `Content-Disposition: attachment; filename=xxx.xlsx` |
| 导出流程 | 先 count 判断，≤200 则全量查询一次性写入 | 避免大数据量导出，接口先返回符合条件的记录数量，前端判断后再执行导出 |
| 文件命名 | `分类记录_{日期}_{时间}.xlsx` | 如 `分类记录_2026-04-28_143052.xlsx` |
| 按钮交互 | 二次点击确认 + 导出期间禁用 | 点击后弹窗确认，导出过程中按钮禁用防止重复点击 |

### 5.7 导出流程细化

```
用户点击"导出Excel"
       │
       ▼
弹窗确认："确定导出当前筛选结果？"
       │
       ├─ 取消 → 关闭弹窗
       │
       └─ 确定 → 禁用按钮，调用接口
                    │
                    ▼
              GET /api/record/export-check
              （返回符合条件的记录数量）
                    │
                    ├─ count > 200 → 前端提示"超过200条，请缩小筛选范围"
                    │
                    └─ count ≤ 200 → 继续调用导出接口
                                       │
                                       ▼
                              GET /api/record/export
                              （返回 Excel 文件流）
                                       │
                                       ▼
                              前端触发下载，恢复按钮状态
```

**API 调整**：新增 `/api/record/export-check` 接口用于预检查数量，或 `/api/record/export` 接口返回结构中包含 `count` 字段，前端根据 count 决定是否继续。

**推荐方案**：`/api/record/export` 接口在响应头中返回 `X-Total-Count`，前端根据该值判断：
- 若 `X-Total-Count > 200`：显示提示，不下载
- 若 `X-Total-Count ≤ 200`：继续读取响应体并下载

### 5.8 验收标准

- 点击导出按钮后弹出确认框
- 确认后，若超过200条，前端提示"超过200条，请缩小筛选范围"，不执行导出
- 若≤200条，浏览器下载 Excel 文件，文件名格式为 `分类记录_{日期}_{时间}.xlsx`
- 导出内容与当前筛选条件一致（含摘要关键字筛选）
- Excel 字段完整：序号、专利摘要、预测类别、置信度（百分比格式）、来源、批次ID（空值显示 "-"）、分类时间
- 无数据时给出提示，不导出空文件
- 导出过程中按钮禁用，导出完成后恢复

```
│  │                                                     │    │

│  │  │ 1  │ 一种...  │ XX公司  │2024-01 │                │    │
│  │  └────┴──────────┴────────┴────────┘                │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  [ 开始分类 ]                                               │
│                                                             │
├─────────────────────────────────────────────────────────────┤

│  │  ████████████░░░░░░░░░░░  48/100  48%               │    │

└─────────────────────────────────────────────────────────────┘
```

### 6.3 数据流

```
前端上传Excel
     │
     ▼
Java后端 /api/patent/batch-classify
     │
     ├─ 解析Excel，提取摘要列
     ├─ 生成 batch_id（UUID）
     ├─ 逐条调用 Python /predict
     │     │
     │     ▼
     │  Python服务（已有单条推理能力）
     │     │
     │     ▼
     ├─ 收集所有结果，批量写入 classification_record（source='batch', batch_id）
     │
     ▼
返回 { batchId, total, results[] }
     │
     ▼
前端展示进度 → 完成后跳转历史页（筛选 batch_id）
```

**关于批量推理策略**：采用Java后端逐条调用Python单条推理接口，而非Python端新增批量接口。理由：
- 代码改动最小，复用现有 `/predict` 接口
- 避免大批量请求导致GPU显存溢出
- 便于逐条记录进度和错误处理
- 50条以内的规模，逐条调用延迟可接受

### 6.4 API 设计

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/patent/batch-classify` | 上传文件并执行批量分类 |

**请求**：`multipart/form-data`，包含 `file`（Excel文件）和 `summaryColumn`（摘要列名，默认"摘要"）。

**响应**：

```json
{
  "code": 200,
  "message": "批量分类完成",
  "data": {
    "batchId": "uuid-xxx",
    "total": 100,
    "success": 98,
    "failed": 2,
    "results": [
      {
        "summary": "一种...",
        "predLabel": "医用成像器械",
        "predProbability": 0.8523
      }
    ]
  }
}
```

### 6.5 改动范围

| 层 | 改动点 | 说明 |
